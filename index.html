What is GraphQL and why would we use it?

  • Awesome talk by Pete Duncanson about GraphQL and the community package
  • Shifts the parameter requirements from server side code to the front-end
  • Allows easy access to filtering, sorting, paging etc
- Pete memorably offers the comparison that GraphQL is to traditional API calls what a jukebox is to mixtapes - WHAT is GraphQL: 2 things: - a query l a n g u a g e for requesting data from a server or API - and a r u n t i m e for fulfilling those queries - How does it work & why should we use it? - firstly you d e s c r i b e the shape of your content - create the graph! - you ask only for what you need and get only what you need back - get many resources in the same request - Allows the FE dev to define what parameters are required and update/remove them in the FE code without needed to re-write API calls or do a full deployment - For the example website that has been created, I have used the community Our.Umbraco.GraphQL as a base: all credits go to Rasmus J Pederson & the OffRoadCode crew

Resolving Related Links


[DefaultGraphQLValueResolver]
public class RelatedLinksValueResolver : GraphQLValueResolver
{
  public override Type GetGraphQLType(PublishedPropertyType type)
  {
	return typeof(ListGraphType< RelatedLinksType >);
  }

  public override bool IsResolver(PublishedPropertyType type)
  {
	return type.PropertyEditorAlias == "Umbraco.RelatedLinks2";
  }
}
- when the package is attempting to resolve a property, it will look through the implemented GraphQLResolver classes to find the right one - here we are passing a l i s t of this RelatedLinksType which is a custom ObjectGraphType (a data "shape")

Related Links Graph Type

public class RelatedLinksType : ObjectGraphType< RelatedLink >
{
  public RelatedLinksType()
  {
	Name = "RelatedLinks2";
	Field< NonNullGraphType< StringGraphType >>(
	  "title",
	  resolve: context => context.Source.Caption);

	Field< NonNullGraphType< StringGraphType >>(
	  "link",
	  resolve: context => context.Source.IsInternal ?
		context.Source.Content.UrlWithDomain() : context.Source.Link);
  }
}
- the RelatedLink is an inbuilt Umbraco Web Model - we are telling it how to map this model to the object graph

Outcome

- and this has been stripped back for brevity, but you can see that the Scene has resolved two links - with their nested properties

Resolving Terratype

- here's a reminder of how the data looked for the Terra Location - coming out as a string, but complex in the database - "datum" is really the bit we're interested in

Terratype Resolver

[DefaultGraphQLValueResolver]
public class TerratypeModelResolver : GraphQLValueResolver
{
	public override Type GetGraphQLType(PublishedPropertyType type)
	{
		return typeof(TerratypeModelType);
	}

	public override bool IsResolver(PublishedPropertyType type)
	{
		return type.PropertyEditorAlias == "Terratype";
	}
}
- Notice that type "Terratype" is the propertyEditorAlias - only one type, because it's not a list

Terratype Graph Type

public class TerratypeModelType : ObjectGraphType< Terratype.Models.Model >
{
  public TerratypeModelType()
  {
	Name = "Terratype";
	Field< NonNullGraphType< FloatGraphType >>(
	  "latitude",
	  resolve: context => context.Source.Position?.ToWgs84().Latitude ?? 0
	);
	Field< NonNullGraphType< FloatGraphType >>(
	  "longitude",
	  resolve: context => context.Source.Position?.ToWgs84().Longitude ?? 0
	);
  }
}
- taking the external Terratype model as input - FloatGraphType rather than the string, but still NonNullGraphType

Terratype resolved

- and we can now feed that into a front-end map or pass it on to another front-end method

Resolving Characters

- so the final bit we're going to look at is how we might go about resolving a custom data type - our weird string... not so beautiful

Custom Data Resolver

public override Type GetGraphQLType(PublishedPropertyType type)
{
	return typeof(ListGraphType< PublishedContentInterfaceGraphType >);
}

public override object Resolve(PublishedPropertyType type, object value)
{
	//this is where the actual work happens
}

public override bool IsResolver(PublishedPropertyType type)
{
	return type.PropertyEditorAlias == "CharacterPicker";
}
- still have IsResolver and still telling it what type we have, but now making use of another method "Resolve" - up until now we've only had to map objects, but now we actually want to change from an ID to a an IPublishedContent object

Convert Ids >> IPublishedContent

var propertyValues = JsonConvert.DeserializeObject< int[] >(value.ToString());
if (propertyValues != null)
{
  var umbracoHelper = new UmbracoHelper(UmbracoContext.Current);
  var nodes = new List< IPublishedContent >();
  foreach (var nodeId in propertyValues)
  {
	var node = umbracoHelper.TypedContent(nodeId);
	if (node != null)
	{
	  nodes.Add(node);
	}
  }
  return nodes;
}
- this is not terribly exciting - we have the value and deserialize it - for each id in our value set we are adding to a list of nodes and returning it as PublishedContentInterfaceGraphType

And the result

- but... it does the trick, as we can now see the published content properties - this means we can access name, characterImage and any other properties we have assigned
